# Hiring Intelligence — ROB Widgets

Three widgets covering the gaps Fred called out in the Apr 29 demo walkthrough. Each is a standalone folder you upload to the Cere Sandbox in ROB.

## Widgets

| Folder | Demo step it serves | Cubby it expects |
|---|---|---|
| `rubric-diff/` | "This is what we fed back to the skill" — trait-weights v3 → v4 + rubric guidance diff + trigger events | `hiring-meta` |
| `score-evolution/` | "Nathan was 8, now 6.5" — same evidence re-scored across rubric versions, threshold line, takeaway | `hiring-scores` |
| `top-signals/` | "Top signals across rounds" — strengths + counter-patterns with evidence + agent attribution | `hiring-traits` |

## Demo flow mapping (Fred's Apr 29 sequence)

```
1. ATS / Notion → candidate updates
2. ROB agent network listening (existing dashboard)
3. Funnel + outcomes (existing dashboard)
4. Top signals for the candidate            ← widgets/top-signals
5. Agent code + execution (existing tab)
6. Outcome: rejected, with reasoning         ← widgets/score-evolution
7. Skill update fed back into the rubric    ← widgets/rubric-diff (THE missing piece)
```

The closing line lands on `rubric-diff` — that is the "is this just one time, or does this intelligence compound?" answer Fred wants.

## Local preview

```bash
cd widgets/rubric-diff
npx serve .
# open http://localhost:3000/dev.html
```

`dev.html` loads `sandbox-injector.js` first to stub `window.WidgetSandbox` + `window.WidgetRuntime`. `index.html` is the production version (no injector) — that is what gets uploaded.

## Demo upload — using inline SQL (no SQLite cubby projection needed)

Each widget folder contains a `manifest.demo.json` (generated by `_build-demo-manifests.mjs`) that inlines the `data.json` payload directly into the SQL as a string literal — `SELECT '<json>' AS payload`. This sidesteps needing a SQLite cubby table to exist.

To use it for the demo:

1. Replace `REPLACE_WITH_AGENT_SERVICE_PUBKEY` / `WORKSPACE_ID` / `STREAM_ID` in `manifest.demo.json`. Set `sqliteCubby.cubbyId` to any SQLite cubby attached to the workspace (it's referenced but never read because the SQL is a constant).
2. **Rename** `manifest.demo.json` → `manifest.json` (the Sandbox only looks for `manifest.json`).
3. Upload the four files: `index.html`, `main.js`, `styles.css`, `manifest.json`.

To regenerate after editing `data.json`: `node widgets/_build-demo-manifests.mjs`.

## Upload to ROB Sandbox

Per `CEF-AI/skills` widget contract:

1. **Folder selection** — upload exactly these files per widget: `index.html`, `main.js`, `styles.css`, `manifest.json`. **Do not upload** `dev.html`, `sandbox-injector.js`, `data.json`, or `README.md`.
2. **Attach query** — pencil icon → Attached query. The platform will use the query in `manifest.json` if it validates against your workspace.
3. **Replace placeholders** — `manifest.json` ships with `REPLACE_WITH_AGENT_SERVICE_PUBKEY`, `REPLACE_WITH_WORKSPACE_ID`, `REPLACE_WITH_STREAM_ID`. Set these to your hiring workspace's values before uploading. Today's HiringPipeline uses Redis-Stack JSON cubbies (`hiring-traits`, `hiring-scores`, `hiring-meta`) — the widget runtime queries SQLite cubbies via SQL, so a SQLite mirror needs to exist or the SQL needs to point at the equivalent SQLite cubby for the workspace.
4. **Preview** — Sandbox UI iframe.
5. **Publish** — embed URL is returned. The widget renders inside ROB.

## Data shape contract (what each widget reads)

All three widgets expect a single-row result with one column called `payload` containing a JSON blob. This keeps the SQL trivial and the schema-shaping responsibility on whatever job populates the SQLite cubby.

### `rubric-diff` payload

```json
{
  "role": "AI Engineer",
  "role_key": "ai-engineer",
  "versions": { "previous": "v3", "current": "v4" },
  "feedback_count": 3,
  "window": "Last 14 days · 9 candidates re-scored.",
  "traits": [
    { "name": "hard_things_done", "old": 0.20, "new": 0.27 }
  ],
  "rubric_diff": [
    { "op": "context|add|remove|modify", "text": "..." }
  ],
  "triggers": [
    { "who": "Sergey", "what": "marked Nathan 6.5...", "when": "Apr 24" }
  ]
}
```

### `score-evolution` payload

```json
{
  "candidate": { "id": "cand_nathan_k", "name": "Nathan K.", "role": "AI Engineer" },
  "verdict": "rejected|accepted|in review",
  "reject_threshold": 70,
  "subnote": "...",
  "history": [
    { "version": "v1", "score": 82, "note": "..." }
  ],
  "takeaway": "..."
}
```

### `top-signals` payload

```json
{
  "candidate": { "id": "cand_nathan_k", "name": "Nathan K.", "role": "AI Engineer" },

    { "trait": "hackathons", "rating": 9.0, "evidence": ["..."], "surfaced_by": "TraitExtractor", "source": "..." }

```

## What's stubbed for the demo

- `data.json` in each widget contains realistic mock payloads matching `hiring-traits` / `hiring-scores` / `hiring-meta` shapes. The Nathan narrative is consistent across all three: rated 8.2 by v1, ends at 6.5 under v4 because hackathons were de-emphasized and `hard_things_done` weight rose.
- The widgets themselves contain zero placeholder text — they pull everything from the JSON payload. Swap mock for real once a SQLite cubby populates the same shape.
- `manifest.json` SQL is `SELECT payload FROM <table> WHERE <key> = ?` against tables (`rubric_versions`, `candidate_score_history`, `candidate_top_signals`) that don't yet exist in `hiring-meta`/`hiring-scores`/`hiring-traits` SQLite mirrors. A small projection job (Distillation agent already runs) can populate them.

## Why these three (and not more)

Fred's three live demo pain points:

1. **"Show me the skill update"** — `rubric-diff`. Anchors the compound-learning narrative.
2. **"Don't just say rejected — show why this was a different verdict than v1 would have given"** — `score-evolution`. Bridges agent execution to outcome with the exact "8 → 6.5" framing.
3. **"Top signals across rounds, tight"** — `top-signals`. Replaces the dashboard's scrolling section with a one-glance widget.

Agent network, funnel, and outcome views already exist in the dashboard and don't need to be re-built as widgets for this demo cycle.
