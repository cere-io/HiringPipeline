# Data Model: Candidate Traits

This document outlines the schema design for `TraitSignal`, which represents the extracted and anonymized metadata for a candidate.

## Entities

### `TraitSignal` (Persisted in `hiring-traits` Cubby)

This object is generated by the Trait Extractor agent parsing the candidate's incoming resume and LinkedIn data.

| Field | Type | Description | PII Risk |
|-------|------|-------------|----------|
| `skills` | `string[]` | Extracted technical and soft skills (e.g., `["Go", "TypeScript", "Systems Design"]`). | None |
| `years_of_experience` | `number` | Total calculated years of professional experience. | None |
| `company_stages` | `string[]` | Categorical descriptors of previous companies (e.g., `["startup", "series_b", "public"]`). | None |
| `education_level` | `string` | Highest degree attained (e.g., `"Bachelors"`, `"Masters"`, `"PhD"`, `"None"`). | None |
| `source_completeness` | `object` | Flags indicating the availability of data (`{ has_resume: boolean, has_linkedin: boolean }`). | None |
| `extracted_at` | `string` | ISO 8601 timestamp of extraction. | None |

## Validation Rules

1. **PII Exclusion**: The schema strictly drops any fields not explicitly defined above. `name`, `email`, `phone`, `location`, and `resume_text` MUST be excluded.
2. **Missing Fields**: If an attribute cannot be determined (e.g., education level is missing from a LinkedIn profile), it defaults to `null` or a sensible empty state (e.g., `[]` for skills).
3. **Data Types**: `years_of_experience` must be an integer >= 0.

## Relationships

- `candidate_id` acts as the primary key in the `hiring-traits` Cubby and links back to the original `CandidateRecord` stored in Notion.
